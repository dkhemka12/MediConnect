MEDICONNECT PROJECT WORKFLOW (BACKEND FOCUS)

============================================================
1) PROJECT OVERVIEW
============================================================
MediConnect is a doctor appointment platform.

Main modules:
- Client (React + Vite): UI, routes, forms, API calls.
- Server (Node.js + Express): auth, users, appointments, access control.
- Database (MongoDB + Mongoose): stores users and appointments.

Backend is the core part of this project.
Frontend sends requests, backend validates and processes everything.


============================================================
2) FOLDER RESPONSIBILITY (IMPORTANT)
============================================================
Root folder:
- Runs client and server together using concurrently.

server/:
- server.js: app entry point, middleware, route mounting, health route.
- config/db.js: MongoDB connection.
- routes/: endpoint mapping.
- controllers/: business logic.
- models/: MongoDB schemas.
- middleware/: token auth protection.
- utils/generateToken.js: JWT creation.

client/:
- pages and components for screens.
- services/ for calling backend APIs.
- auth data is kept in localStorage.


============================================================
3) RUNTIME REQUIREMENTS
============================================================
Software:
- Node.js (recommended modern LTS)
- npm
- MongoDB database (local or cloud)

Environment variables used by backend:
- MONGO_URI               (MongoDB connection string)
- JWT_SECRET              (secret key for JWT signing)
- JWT_EXPIRES_IN          (optional, default is 7d)
- CLIENT_URL              (optional, default http://localhost:5173)
- PORT                    (optional, default 5000)

Environment variable used by frontend:
- VITE_API_BASE_URL       (optional, default http://localhost:5000/api)

Install and run:
1. In root: npm install
2. In server: npm install
3. In client: npm install
4. In root: npm run dev

This starts both:
- Backend on port 5000 (by default)
- Frontend on port 5173 (Vite default)


============================================================
4) BACKEND STARTUP FLOW
============================================================
Step 1:
- server/server.js loads env values with dotenv.

Step 2:
- Express app is created.
- CORS is enabled with allowed origin from CLIENT_URL.
- JSON body parser is enabled with express.json().

Step 3:
- Routes are mounted:
  - /api/auth
  - /api/users
  - /api/appointments

Step 4:
- /api/health route returns server status.

Step 5:
- startServer() calls connectDB().
- Database connection is established first.
- Server starts listening only after DB is connected.


============================================================
5) DATABASE MODELS
============================================================
A) User model
Fields:
- name (required)
- email (required, unique, lowercase)
- password (required, min length 6)
- role (patient | doctor | admin, default patient)
- phone (optional)
- isActive (default true)
- createdAt, updatedAt (timestamps)

B) Appointment model
Fields:
- patientId (ObjectId ref User)
- doctorId (ObjectId ref User)
- date (string)
- time (string)
- status (pending | approved | cancelled, default pending)
- createdAt, updatedAt (timestamps)


============================================================
6) AUTHENTICATION FLOW (JWT)
============================================================
Register flow:
1. Client sends POST /api/auth/register with name, email, password, role, phone.
2. Backend normalizes email to lowercase and trimmed format.
3. Validates required fields.
4. Checks duplicate email in DB.
5. Hashes password using bcryptjs.
6. Creates user.
7. Generates JWT token with userId and role.
8. Returns token + safe user data.

Login flow:
1. Client sends POST /api/auth/login with email + password.
2. Backend normalizes email.
3. Finds user by email.
4. Compares password using bcryptjs.compare().
5. Generates JWT token.
6. Returns token + user profile fields.

Token structure:
- Payload: userId, role
- Signed with JWT_SECRET
- Expiry from JWT_EXPIRES_IN (or 7d)

Client session storage:
- Token is saved in localStorage under key: token
- Role is saved in localStorage under key: role
- For authenticated requests, client sends:
  Authorization: Bearer <token>


============================================================
7) AUTHORIZATION / ACCESS CONTROL FLOW
============================================================
protect middleware (server/middleware/authMiddleware.js):
1. Reads Authorization header.
2. Extracts Bearer token.
3. Verifies token with JWT_SECRET.
4. Fetches user from DB and removes password field.
5. Attaches user object to req.user.
6. If anything fails -> returns 401 Not authorized.

Where it is used:
- All appointment routes use router.use(protect),
  so all appointment endpoints are private.

Note:
- roleMiddleware.js currently has no active code.
- Role filtering for appointments is handled in controller logic.


============================================================
8) APPOINTMENT WORKFLOW (CORE BUSINESS FLOW)
============================================================
A) Book appointment
Endpoint: POST /api/appointments
Access: Private
Flow:
1. User must be logged in (token required).
2. Backend validates doctorId, date, time.
3. Creates appointment with patientId from req.user._id.
4. Default status is pending.
5. Returns created appointment.

B) Get appointments
Endpoint: GET /api/appointments
Access: Private
Flow:
1. Backend checks req.user.role.
2. If patient -> only own appointments.
3. If doctor -> only appointments assigned to that doctor.
4. If admin -> all appointments.
5. Populates patient and doctor basic info (name, email, phone).
6. Sorts latest first.
7. Returns list.

C) Update appointment status
Endpoint: PUT /api/appointments/:id
Access: Private
Flow:
1. Validates status in request body.
2. Updates appointment status by id.
3. Returns updated record.
4. If id not found -> 404.

9) USER DATA WORKFLOW
============================================================
Endpoints:

- GET /api/users/patients

- Returns users without password field.

Important note:

- If needed for production, add protect and role checks.


============================================================
10) FRONTEND -> BACKEND REQUEST PIPELINE (SHORT)
============================================================
1. UI page triggers service function.
2. Service calls requestJson() from client/services/api.js.
3. requestJson() attaches token automatically if available.
4. Request hits backend route.
5. Route calls controller.
6. Controller reads/writes MongoDB.
7. JSON response returns to client.
8. UI updates state.

Auth guard in frontend:
- RequireAuth component checks token in localStorage.
- If no token, user is redirected to /login.


============================================================
11) ERROR HANDLING STRATEGY
============================================================
Backend response style:
- success: true/false
- message: readable message
- data/count when needed

Typical status codes:
- 200: success fetch/update/login
- 201: created (register/appointment)
- 400: bad input
- 401: unauthorized or invalid credentials
- 404: record not found
- 409: duplicate email
- 500: internal server error

Client behavior:
- requestJson throws Error(message) for non-2xx responses.
- Some appointment service methods have localStorage fallback if API fails.

15) ONE-LINE FULL FLOW
============================================================
User logs in -> gets JWT -> client stores token -> protected API calls include Bearer token -> backend verifies token -> controller executes business rules -> MongoDB stores/retrieves data -> response returns -> UI updates.
